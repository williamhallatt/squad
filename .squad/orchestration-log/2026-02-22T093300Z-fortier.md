## Work Done

- Added `sendMessage()` and `closeSession()` OTel spans to SquadClient with `squad.session.message` and `squad.session.stream` child spans
- Implemented `StreamingPipeline.markMessageStart(sessionId)` as explicit latency tracking entry point
- Wired TTFT, response duration, and throughput metrics into StreamingPipeline
- Records TTFT on first `message_delta` with `index === 0`
- Records duration and tokens/sec when `usage` event arrives
- Auto-cleans tracking state after usage event or `clear()`
- 15 new tests added to test suite

## Key Decisions

1. **Explicit entry point:** `markMessageStart()` requires opt-in before sending — callers who don't need latency metrics (tests, offline replay) pay zero overhead
2. **Pipeline doesn't own send:** Latency tracking lives in pipeline, not SquadClient — separation of concerns prevents hidden coupling
3. **Metrics recorded after user handlers:** User handlers see events before OTel instrumentation; handler failures don't block metrics
